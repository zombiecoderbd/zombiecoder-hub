এই ডকুমেন্টটি একটি লিখিত প্রতিশ্রুতি। এখানে প্রতিটি সমস্যার জন্য:

  • সমস্যা কোথায় (exact file, exact line)
  • সমাধান কি (exact code change)
  • আউটপুট কি আসবে (exact expected result)
  • আমি কতটুকু confident (percentage)

তিনটি সম্ভাবনা:

পরিস্থিতিচিহ্নিতকরণ
কাজ সম্পূর্ণ হয়, সব fix কার্যকর, build pass করে Audit Verified - রিপোর্ট ও execution মিলেছে
কিছু fix কাজ করে, কিছু করে না (genuine technical obstacle) Partial Delivery - কোনটি ব্যর্থ হলো এবং কেন সেটি transparently জানাবো। এটি গ্রহণযোগ্য যদি ব্যর্থতার কারণ যুক্তিসঙ্গত হয়।
অডিট রিপোর্টে বলেছি A সমস্যা, কিন্তু আসলে A সমস্যা নয়। অথবা বলেছি fix করব, কিন্তু ভিন্ন কিছু করেছি। অথবা "100% confident" বলে fail করেছি কিন্তু কারণ জানাইনি। Audit Bypass / False Report - বিশ্লেষণে মিথ্যা তথ্য দেওয়া হয়েছে অথবা প্রতিশ্রুতি পালন না করে ঢাকার চেষ্টা করা হয়েছে। এটি "কর্পোরেট দালালি" - corporate safety script দিয়ে আসল সমস্যা ঢাকা।

প্রতিটি সমস্যা fix-এর পর verifiable প্রমাণ দেওয়া হবে:

  • Schema change -> migration command output
  • Code change -> file diff (আগে vs পরে)
  • Build fix -> next build output (pass/fail)
  • WebSocket fix -> test script output

যদি "100% confident" বলে কোনো item fail করে এবং গ্রহণযোগ্য কোনো কারণ উপস্থাপন না করতে পারি, সেটি "Audit Bypass" হিসেবে চিহ্নিত হবে। সেটিই সত্য।
